Таблица предпросмотра «ТаблицаДанных» — что убрать и что добавить (этап 6)
================================================================================

1) Удалить неиспользуемые колонки (в конфигураторе / EDT)
---------------------------------------------------------
В реквизите формы «ТаблицаДанных» (тип ТаблицаЗначений) удалите колонки:
  • Соглашение
  • Договор
  • ДоговорКонтрагента

На закладке элементов формы в табличном поле удалите соответствующие дочерние
поля (иначе останутся «битые» привязки к данным).

Код обработки эти колонки не заполняет и не читает.


2) Добавить колонку «Порядок расчётов» для предпросмотра
---------------------------------------------------------
Имя колонки в таблице (как в коде BSL):  ПорядокРасчетов

Синоним (подпись в форме):              Порядок расчётов
   (или «Порядок расчетов» — как принято у вас в интерфейсе)

Тип значения:  ПеречислениеСсылка.ПорядокРасчетов

В конфигураторе: при добавлении колонки к реквизиту «ТаблицаДанных» выберите тип
«Перечисление ссылка» → перечисление «ПорядокРасчетов» (имя в метаданных УТ).

После этого в таблице будет видно текущее значение перечисления по каждой
строке (как в документе). Код в Код1С_УТ_06_... заполняет колонку из запроса.


3) Старая колонка «ТекущийПорядокРасчетов» (Строка)
---------------------------------------------------
Раньше в колонку с таким именем записывалось только ПРЕДСТАВЛЕНИЕ перечисления.
Код по-прежнему пытается заполнить её для совместимости со старыми формами.

Если добавили «ПорядокРасчетов» и дублирование не нужно — колонку
«ТекущийПорядокРасчетов» можно удалить из таблицы и с формы; на работу
обработки это не влияет.


4) Важно про отбор предпросмотра
--------------------------------
Запрос загрузки по умолчанию отбирает только реализации, у которых
ПорядокРасчетов <> ПоДоговорамКонтрагентов (или только «По расчетным документам»,
если на форме включён узкий отбор).

То есть документы, у которых «уже всё хорошо» (уже ПоДоговорамКонтрагентов),
в список предпросмотра НЕ попадают — их не нужно отсеивать вручную.

Колонка «ПорядокРасчетов» нужна, чтобы внутри отобранных строк сразу видеть,
какое сейчас «нецелевое» значение (ПоРасчетнымДокументам, ПоНакладным и т.д.)
без открытия карточки.

Если понадобится видеть в одной таблице и «хорошие», и «плохие» — это
отдельное изменение условия в тексте запроса в процедуре
ЗагрузитьПредпросмотрНаСервере.


5) Опциональные реквизиты формы (не колонки таблицы)
----------------------------------------------------
Имя: ТолькоПоРасчетнымДокументам   Тип: Булево   Смысл: узкий отбор в предпросмотре
      только ПорядокРасчетов = ПоРасчетнымДокументам. Без реквизита — широкий отбор.

Имя: НеСинхронизироватьЗаказКлиента   Тип: Булево   Смысл: если Истина — заказ клиента
      вообще не читается и не записывается (только реализация). Удобно, если заказы
      трогать нельзя по политике.

Имя: НеСинхронизироватьЗаказПоставщику   Тип: Булево   Смысл: в режиме приобретения —
      не синхронизировать ПорядокРасчетов в заказе поставщику (аналог флага по заказу клиента).

Режим документов (два флажка на форме, имена как в коде модуля):
  • РежимРеализацияТоваровУслуг   Тип: Булево   Подпись, например: «Реализация товаров и услуг»
  • РежимПриобретениеТоваровУслуг Тип: Булево   Подпись, например: «Приобретение товаров и услуг»
  В один момент времени должна быть отмечена ровно одна галочка. Если отмечены обе —
  предпросмотр и применение не запускаются (сообщение пользователю).
  На событие «ПриИзменении» полей привяжите процедуры модуля формы:
  РежимРеализацияТоваровУслугПриИзменении и РежимПриобретениеТоваровУслугПриИзменении
  (они снимают вторую галочку при включении первой).
  Значения по умолчанию в конфигураторе: РежимРеализацияТоваровУслуг = Истина,
  РежимПриобретениеТоваровУслуг = Ложь.

Колонка таблицы «Реализация» (одно поле для обоих видов):
  • В ячейку пишется ссылка и на Документ.РеализацияТоваровУслуг, и на Документ.ПриобретениеТоваровУслуг.
  • Тип колонки в форме задайте составным: оба вида документов. Иначе ссылка не сохранится,
    будет пустая ячейка и ошибка «Элемент не выбран!» на ПолучитьОбъект.

Опциональная колонка «ЭтоПриобретение» (Булево):
  • При загрузке предпросмотра код сам выставляет: Истина в режиме приобретения, Ложь в режиме реализаций.
  • При исправлении по строке выбирается синхронизация заказа поставщику vs заказа клиента по ЭТОМУ
    признаку; если колонки нет — берётся текущий флажок режима на форме (как раньше).
  • Имеет смысл добавить, если после загрузки таблицы вы можете переключить режим на форме,
    но всё равно обрабатывать уже загруженные строки — тогда ветка по заказу останется верной по строке.

Альтернатива по ссылке: колонка «Документ» (составной тип: реализация + приобретение) —
код при загрузке дублирует в неё ту же ссылку и при обработке читает ссылку из «Реализация»
или из «Документ».

Колонка «Заказ» (если есть): для приобретения заполняется из ЗаказПоставщику; тип колонки
должен допускать ссылку на заказ поставщику (или оставьте только для реализаций — тогда
для приобретений колонка просто не заполнится).

По умолчанию реквизиты режима и флаги синхронизации заказов можно не создавать
(будет режим реализаций, как раньше).

6) Команды «Выбрать все» / «Снять все» и фильтр по таблице
----------------------------------------------------------
Код на клиенте отмечает или снимает галочку «Выбрать» только у строк, которые
проходят пользовательский отбор/поиск по таблице (метод элемента ТаблицаФормы
«ПроверитьСтроку»). Для реквизита типа «Таблица значений» в метод передаётся
идентификатор строки: Строка.ПолучитьИдентификатор() (при ошибке — Индекс строки
в ТаблицаДанных), иначе отбор визуально не совпадает с фактом выбора всех строк.

Поиск элемента таблицы: сначала по имени «ТаблицаДанных»,
при необходимости по ПутьКДанным (чтение ПутьКДанным в Попытка — у части
элементов/клиентов свойство может быть недоступно). Обход включает
подчинённые элементы (таблица внутри группы/страницы). Если элемент не найден
— команды работают по всем строкам и выводится подсказка.

Поведение синхронизации заказа (без флага «НеСинхронизироватьЗаказКлиента»):
сначала пытаемся выставить ПорядокРасчетов в заказе и провести заказ; если УТ
запрещает (объект расчётов, остатки по заказу и т.п.) — ошибка по заказу
НЕ отменяет транзакцию: реализация всё равно перепроводится с целевым порядком,
в комментарии строки будет «OK; заказ не меняли: …». В лог пишется ПРЕДУПРЕЖДЕНИЕ.

Этап 4 не «обходил» платформу: там тоже Заказ.Записать(Проведение) до реализации;
при тех же данных возможны те же отказы по заказу.
