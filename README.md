# Проект восстановления данных в 1С УТ

## О чем проект

Проект восстанавливает корректный контур взаиморасчетов и движения денег в `1С:Управление торговлей` на основе исторически корректных данных из `1С:Бухгалтерия предприятия`.

Основная идея:

* `1С БП` используется как источник истины по платежам;
* `1С УТ` приводится в состояние, где корректно отражены контрагенты, договоры и банковские документы;
* после загрузки данные в УТ позволяют достоверно анализировать ДЗ/КЗ и денежные потоки.

## Начальная точка

На старте в УТ были проблемы с качеством данных:

* часть контрагентов и реквизитов отсутствовала или была заполнена неполно;
* в документах реализации не везде были корректно проставлены соглашения и договоры;
* карточки ИП/ФЛ требовали нормализации;
* банковские платежи в УТ были неполными или некорректными относительно БП.

## Конечная цель

Получить в УТ рабочий и проверяемый слой данных:

* контрагенты приведены к корректному виду;
* договорная аналитика согласована для корректного учета;
* платежки из БП загружены в УТ без дублей (по УИД);
* оставшиеся сложные случаи выделены в ручной контроль.

## Пошаговый маршрут

### Этап 1. Подготовка клиентов для внесения в УТ

* Выгружаются списки клиентов из БП и УТ.
* Python-скрипт сравнивает данные, находит отсутствующих в УТ клиентов и дополняет банковские реквизиты.
* На выходе: подготовленный файл для последующей загрузки/дозаполнения.

### Этап 2. Нормализация ИП и ФЛ

* Python-скрипт обогащает данные по ИНН через сервис ФНС (ЕГРЮЛ/ЕГРИП).
* Внешняя обработка 1С обновляет карточки контрагентов/партнеров в УТ по коду.
* Результат: единый, валидный формат карточек.

### Этап 3. Создание договоров

* Подготавливаются данные для массового заведения договоров.
* Этап закрывает разрывы, мешающие корректной загрузке платежей и аналитике расчетов.

### Этап 4. Дозаполнение цепочек реализации

* Внешняя обработка 1С массово проставляет `Соглашение` и `Договор` в документах реализации и связанных заказах.
* Применение идет с предпросмотром, выбором строк и безопасными транзакциями.

### Этап 5. Загрузка платежек в УТ

* Python-пайплайн очищает выгрузку банка из БП, применяет маппинг операций и статей ДДС, формирует готовые файлы.
* Внешняя обработка 1С загружает документы `Поступление/Списание`, проверяет дубли по УИД и фиксирует связи в регистре.
* По итогам обеспечивается перенос истории платежей в УТ.

### Этап 99. Ручной контроль

* Сюда попадают случаи, которые нельзя надежно обработать автоматически.
* Такие платежи проверяются и заносятся вручную, чтобы итоговый остаток и аналитика полностью сошлись.

## Ключевые артефакты

* Python: подготовка и нормализация данных (`.py`).
* 1С BSL: массовые обновления и загрузка в УТ (`.bsl`).
* Текстовые запросы/описания: правила выборки и настройки (`.txt`, `.md`, `.docx`).
* Excel-файлы: входные и выходные таблицы для этапов обработки.

## Текущее состояние по материалам проекта

Из документов и кода видно, что основная логика пайплайна уже реализована:

* подготовка клиентов и нормализация карточек;
* массовое исправление договорной связности;
* подготовка и загрузка платежек с защитой от повторной загрузки;
* выделение ручного контроля для исключений.

Дальнейший фокус - стабильное прогонное тестирование на копии базы и финальный перенос в рабочий контур.
