# Этап 5 - Загружаем платёжки в 1С

## Цель этапа

Перенести банковские платежи из `1С БП` в `1С УТ` так, чтобы:
- документы в УТ создавались в корректном виде (`Поступление`/`Списание`);
- сохранялась аналитика по контрагентам, договорам и статьям ДДС;
- исключалась повторная загрузка одних и тех же платежей.

## Состав этапа

### Python-подготовка данных
- `03_modefide.py` - основной скрипт подготовки файла к загрузке:
  - очищает и нормализует данные банковской выгрузки;
  - применяет маппинг операций (`mapping_operations_ut.xlsx`);
  - применяет маппинг статей ДДС (`dds_mapping_ut_bp.xlsx`);
  - определяет НДС по текстам назначений и оснований;
  - формирует итоговые файлы (`*_prepared.xlsx`, сводки, нераспознанные операции).
- `04_разбиение данных.py` - разбиение подготовленного файла по периодам (месяц/квартал/полугодие).
- `Эксперемент/Код разбивает файл по дням.py` - экспериментальный вариант разбиения по дням.

### Загрузка в 1С УТ
- `Код_загрузки_в_базу УТ.bsl` - актуальная внешняя обработка загрузки:
  - читает Excel и подбирает объекты УТ (организация, контрагент, счета, статьи ДДС);
  - создает документы оплаты в УТ;
  - проверяет дубли по УИД через регистр `КАА_СвязиДокументовБП`;
  - при необходимости создает договоры и банковские счета;
  - пишет подробный лог выполнения.

### Дополнительные файлы из `src`
- `Материалы из src/Выгрузка_Банк_из_БП.epf` - выгрузка банка из БП.
- `Материалы из src/РеквизитыУТ.epf` - выгрузка реквизитов/метаданных УТ для сопоставления.
- `Материалы из src/ЗагрузкаПлатежекИзБП.epf` - вариант внешней обработки загрузки платежек.
- `Материалы из src/KAA_Bank.epf` - файл, вероятно связанный с банковским контуром (требует финальной проверки внутри 1С).

## Архив и эволюция

Папка `Арихв рабочих весрий` содержит:
- старые версии Python/BSL кода;
- промежуточные текстовые фрагменты;
- служебные выгрузки и тестовые файлы.

Эти материалы важны как история разработки, но для текущей работы опираться нужно на основные файлы этапа, перечисленные выше.

## Результат этапа

На выходе получаем загруженные в УТ платежные документы с прослеживаемой связью с источником из БП и минимальным количеством ручных корректировок.

## Усиления обработки (апрель 2026)

- Ускорен предпросмотр:
  - в цикле разбора Excel включены кэши по ИНН/организациям/счетам/статьям/УИД;
  - повторные строки не запускают одни и те же SQL-запросы заново.
- Усилена защита от дублей:
  - проверка дубля по УИД выполняется не только на предпросмотре, но и повторно перед записью;
  - запись документа и запись связи в `КАА_СвязиДокументовБП` выполняются в одной транзакции.
- Улучшена логика договоров:
  - поиск/создание договора дополнен нормализацией по подходу этапа 3 (дата, номер, статус, партнер, статья ДДС).
- Добавлена обработка счетов-заглушек:
  - если счет отсутствует/выглядит как заглушка/массово повторяется, формируется детерминированный техсчет;
  - техсчет создается уникально по клиенту, чтобы разные клиенты не делили один и тот же номер.
- Добавлен финальный контроль инвариантов:
  - проверка контрагента и суммы до записи и после записи документа;
  - контроль суммы расшифровки платежа (до копейки);
  - при нарушении инвариантов откатывается только текущая строка.
- В блоке `FIFO` переработана логика распределения:
  - выборка долгов по договору;
  - распределение в порядке FIFO;
  - остаток суммы автоматически уходит в аванс в рамках того же договора.

## Baseline и приемка

### Базовые замеры (до/после)

Для сравнения производительности фиксируем по каждому тест-файлу:
- время `Загрузить предпросмотр` (сек);
- время `Загрузить в базу` (сек);
- итог: `Создано/Пропущено/Ошибок`;
- количество строк со статусами `Готов/Дубль/Ошибка`.

Рекомендуемые тестовые файлы:
- малый (`~100-500` строк),
- средний (`~3k-10k` строк),
- большой (`50k+` строк, если позволяет контур).

### Чеклист приемки

- Повторный запуск того же файла не создает новые документы (второй контур дублей работает).
- Для строк с одинаковым счетом-заглушкой у разных клиентов создаются разные техсчета.
- В ссылочные поля не записываются текстовые значения.
- При любой ошибке финального инварианта откатывается только текущая строка.
- Сумма документа и сумма ТЧ `РасшифровкаПлатежа` совпадают до копейки.
- FIFO-распределение не выходит за рамки выбранного договора.
