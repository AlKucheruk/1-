# Этап 5 — Дозаполнение приходов Договором (исходное описание копии)

Ниже — описание **оригинальной** логики этапа 5; в рабочей обработке этапа 7 оно **не** используется.

## Что реализовано

- Добавлена обработка для приходных документов по поставщикам на базе логики этапа 4.
- Сохранены: батчевая обработка, транзакции по одному документу, перепроведение, контроль суммы, логирование и мониторинг прогресса.
- Базовый рабочий файл: `Код1С_УТ_05_Исправление Поступлений.bsl`.

## Контур документов (первая версия)

### Включено сейчас

- `ПриобретениеТоваровУслуг` — основной целевой документ.
- `ПриобретениеУслугПрочихАктивов` — включен как кандидат с runtime-проверкой реквизитов.
- `ПоступлениеТоваровНаСклад` — включен как кандидат с runtime-проверкой реквизитов.
- `КорректировкаПриобретения` — включен как кандидат с runtime-проверкой реквизитов.

### Следующий релиз (после проверки на базе)

- Документы, которые не проходят по структуре реквизитов/поведению при перепроведении в конкретной УТ-базе.
- Сценарии с нестандартными связями оснований, где `Договор` хранится не в самом документе, а только в связанных объектах.
- Дополнительные приходные типы, подтвержденные по фактическим данным эксплуатации.

## Гибридная валидация реквизитов (вариант C)

Валидация сделана в двух слоях:

1. **Гипотеза по коду**: обработка ожидает реквизиты `Контрагент`, `СуммаДокумента`, `Договор`, опционально `ЗаказПоставщику`.
2. **Проверка в runtime**: если документ или реквизит отсутствует, тип документа пропускается с записью причины в лог.

Технически это реализовано так:

- Запросы формируются отдельно по каждому типу документа-кандидата.
- Выполнение каждого запроса защищено `Попытка/Исключение`.
- Установка `Договор` выполняется безопасной функцией через `Выполнить(...)`, чтобы не падать на документах с иной моделью реквизитов.
- Связанный заказ обновляется только если найден и если в нем есть `Договор`.

Проверки по реквизитам в коде:

- `Контрагент` и `СуммаДокумента` читаются из запроса по документу.
- `Договор`:
  - читается в предпросмотр;
  - при пустом значении берется из справочника договоров контрагента (тип `СПоставщиком`);
  - записывается в документ только через безопасную установку свойства.
- `ЗаказПоставщику`:
  - читается как опциональная ссылка;
  - при наличии выполняется попытка установки `Договор` и перепроведения заказа.

## Правила безопасной записи

- Обработка идет по одному документу в транзакции.
- Для уже заполненных документов выставляется явный пропуск.
- Для ошибок фиксируется причина и обработка продолжает следующий документ.
- После записи выполняется перепроведение.
- Контролируется неизменность `СуммаДокумента` до/после записи.

## Протокол тестирования (для запуска в 1С)

### Минимальный прогон

1. Выбрать диапазон дат с 10–20 документами прихода.
2. Запустить предпросмотр с фильтром по одному контрагенту.
3. Отметить все строки и выполнить обработку.
4. Проверить в логах: количество успехов, пропусков и ошибок.

### Смешанный прогон

1. Сформировать выборку, где часть документов уже с договором, часть без.
2. Повторить запуск и убедиться, что:
   - заполненные документы пропускаются;
   - пустые получают договор;
   - при ошибках обработка не останавливается полностью.

### Проверки консистентности

1. Сверить `СуммаДокумента` до/после на случайной подвыборке.
2. Проверить перепроведение и отсутствие критичных изменений движений.
3. Перезапустить обработку на той же выборке и подтвердить идемпотентность.

## Что смотреть в логе

- Старт/итоги по каждому батчу.
- Пропуски по документам без необходимости обновления.
- Ошибки по документам с указанием ссылки и контрагента.
- Сообщения о пропуске типов документов, которых нет в конкретной конфигурации.
